Egy context, ami a mögöttes logikát építi ki:

import React, { createContext, useState, useEffect } from "react";
import { axiosInstance } from "../ApiContext";

export const ProjektContext = createContext();

export const ProjektProvider = ({ children }) => {
    const [posts, setPosts] = useState([]);
    const [loading, setLoading] = useState(true);

    // GET - Adatok lekérése a Laravel backendtől
    const getPosts = async () => {
        try {
            const response = await axiosInstance.get("/posts");
            // Laravel-nél gyakran response.data vagy response.data.data a tömb
            const adatok = Array.isArray(response.data) ? response.data : response.data.data;
            setPosts(adatok || []);
        } catch (error) {
            console.error("Hiba a letöltésnél:", error);
            setPosts([]); 
        } finally {
            setLoading(false);
        }
    };

    // POST - Új bejegyzés küldése (store metódus a backendben)
    const addPost = async (ujAdat) => {
        try {
            await axiosInstance.post("/posts", ujAdat);
            getPosts(); // Frissítjük a listát a sikeres mentés után
        } catch (error) {
            console.error("Sikertelen mentés:", error);
        }
    };

    useEffect(() => {
        getPosts();
    }, []);

    return (
        <ProjektContext.Provider value={{ posts, loading, addPost }}>
            {children}
        </ProjektContext.Provider>
    );
};

import { createContext, useState, useEffect } from "react";

export const ApiContext = createContext();

export const ApiProvider = ({ children }) => {
    const [szakdogaLista, setSzakdogaLista] = useState([]);

    const getAdatok = async () => {
        try {
            const response = await axiosInstance.get("/szakdogak");
            setSzakdogaLista(response.data);
        } catch (error) {
            console.error("Hiba a lekérésnél:", error);
        }
    };

    const adatUj = async (adat) => {
        try {
            await axiosInstance.post("/szakdogak", adat);
            getAdatok(); 
        } catch (error) {
            console.error("Hiba a mentésnél:", error);
        }
    };

    const adatTorol = async (id) => {
        try {
            await axiosInstance.delete(`/szakdogak/${id}`);
            getAdatok(); 
        } catch (error) {
            console.error("Hiba a törlésnél:", error);
        }
    };

    const adatModosit = async (id, adat) => {
        try {
            await axiosInstance.put(`/szakdogak/${id}`, adat);
            getAdatok(); 
        } catch (error) {
            console.error("Hiba a módosításnál:", error);
        }
    };

    useEffect(() => {
        getAdatok();
    }, []);

    return (
        <ApiContext.Provider value={{ szakdogaLista, adatUj, adatTorol, adatModosit }}>

4. lépés: néhány fogalom, mely segíthet a vizsgán

Alapfogalmak
* Komponens: A React alkalmazás alapvető építőköve, egy önálló, újrafelhasználható kódegység (függvény), amely a felhasználói felület egy részéért és annak megjelenítéséért felelős.
* Props: A komponenseknek kívülről átadott paraméterek (tulajdonságok), amelyek segítségével adatokat küldhetünk a szülő komponenstől a gyerek komponens felé; értékeük csak olvasható.
* State (Állapot): A komponensen belüli dinamikus adat, amely meghatározza a komponens aktuális viselkedését és kinézetét; ha az értéke megváltozik, a React automatikusan újrarendereli a felületet.
* Hook: Speciális függvények, amelyek lehetővé teszik a React állapotkezelési és életciklus-funkcióinak használatát funkcionális komponensekben anélkül, hogy osztályokat (class) kellene írnunk.
* useState hook: A leggyakrabban használt hook, amely egy állapotváltozót és egy annak frissítésére szolgáló függvényt ad vissza egy tömbben, kezdőérték megadása mellett.

Adatátvitel és Külső Kapcsolatok
* Context & Provider: A Context egy módszer az adatok globális megosztására a komponensfán keresztül "prop-drilling" (szinteken átívelő kézi továbbítás) nélkül. A Provider az a komponens, amely körbeöleli az alkalmazást (vagy egy részét), és elérhetővé teszi a benne tárolt value értéket az összes leszármazott számára.
* useContext hook: Ezzel a hook-kal tudunk egy adott komponensen belül egyszerűen hozzáférni egy korábban létrehozott Context aktuális értékéhez.
* RESTful alkalmazás: Olyan szoftverarchitektúra, amely a HTTP protokoll szabványait (GET, POST, PUT, DELETE) használja az erőforrások elérésére és módosítására, általában JSON formátumban.
* Asszinkron hívás: Olyan művelet (pl. adatlekérés a szerverről), amely nem blokkolja a program futását; a kérés elindul a háttérben, és a válasz megérkezésekor fut le a hozzárendelt logika.
* Végpont (Endpoint): Egy specifikus webcím (URL) az API-n belül, ahol egy adott erőforrás elérhető vagy egy konkrét művelet végrehajtható.
* HTTP kérés: Üzenetváltás a kliens (böngésző) és a szerver között, amely tartalmazza a metódust (mit akarunk csinálni), az URL-t és opcionálisan adatokat (body).

Logika és Navigáció
* useEffect: Mellékhatások (pl. API hívások, feliratkozások, manuális DOM módosítás) kezelésére szolgáló hook, amely a komponens renderelése után fut le.
* useNavigate: A react-router-dom könyvtár hook-ja, amellyel programozottan (kódból, pl. egy gombnyomásra vagy sikeres mentés után) tudjuk átirányítani a felhasználót egy másik útvonalra.
* Űrlap, mint saját állapottal rendelkező elem: Reactben az űrlapmezők (inputok) értékét általában a React state-je vezérli, így az input aktuális értéke mindig szinkronban van a komponens belső állapotával (controlled component).
* Űrlap működéséhez szükséges elemek: Kell egy state az adatok tárolásához, a mező value attribútuma ezt az állapotot mutatja, az onChange esemény frissíti az állapotot gépeléskor, az onSubmit pedig kezeli az adatok beküldését (gyakran event.preventDefault() használatával a frissítés megelőzésére).
* Navigáció és routing készítése: A react-router-dom használatával az alkalmazást egy BrowserRouter-be csomagoljuk, a Routes és Route komponensekkel definiáljuk az útvonal-alkatrész párokat, a navigációhoz pedig a Link komponenst vagy a useNavigate hook-ot használjuk.
1. Token-alapú hitelesítés (pl. JWT - JSON Web Token)
Mire való? Azonosításra és jogosultságkezelésre (Ki vagy te?).
* Működése: Amikor bejelentkezel, a szerver ad neked egy tokent (mint egy digitális belépőkártyát). Ezt a tokent a böngésző elmenti (LocalStorage-ba vagy sütibe), és minden egyes kérésnél elküldi a szervernek (általában az Authorization: Bearer <token> fejlécben).
* Célja: Kiváltja a hagyományos, szerveroldali munkamenet-tárolást (session). A szervernek nem kell adatbázisban keresgélnie, a token önmagában igazolja a kilétedet.
2. CSRF Token (Cross-Site Request Forgery elleni védelem)
Mire való? A felhasználó szándékának igazolására (Valóban te akartad ezt a műveletet?).
* Működése: A szerver generál egy egyedi, véletlenszerű kódot minden munkamenethez vagy űrlaphoz. Ezt a kódot a kliensnek vissza kell küldenie a szervernek (pl. egy rejtett űrlapmezőben vagy egy egyedi fejlécben) minden olyan műveletnél, ami változtat az adatokon (POST, PUT, DELETE).
* Célja: Megakadályozza, hogy egy kártékony weboldal a háttérben – anélkül, hogy tudnál róla – kéréseket küldjön a nevedben (pl. pénzátutalás vagy jelszómódosítás), kihasználva azt, hogy a böngésződ automatikusan küldi a sütiket (cookies).

Mikor melyiket használd?
1. Ha a JWT-t LocalStorage-ban tárolod: Ilyenkor a kéréseket manuálisan, JavaScripttel kell összeállítanod (Authorization header). Mivel a böngésző nem küldi el automatikusan a LocalStorage tartalmát más oldalak kéréseihez, alapvetően védve vagy a CSRF ellen.
2. Ha a JWT-t (vagy Session ID-t) Sütiben (Cookie) tárolod: Ez biztonságosabb az XSS (kódbefecskendezés) ellen, de mivel a böngésző automatikusan csatolja a sütiket minden kéréshez, sebezhetővé válsz a CSRF-el szemben. Ilyenkor kötelező a CSRF token használata is.
